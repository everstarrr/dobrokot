7.1 Технологический стек клиентской части
Клиентская часть платформы реализована на стеке Next.js 15 в режиме App Router поверх React 19 и TypeScript 5.7. Выбор стека обусловлен следующими соображениями.
Next.js 15 обеспечивает гибридный рендеринг: статические страницы (главная, блог, страница результатов) собираются как Server Components на этапе билда и отдаются клиенту в виде готового HTML, а интерактивные участки (формы, личный кабинет, поиск крови, карта доноров) реализуются как Client Components и подгружаются в виде небольших JavaScript-чанков. Это даёт быстрый первый рендер и хорошие показатели Core Web Vitals при минимальном объёме клиентского кода.
React 19 используется в качестве базовой UI-библиотеки; новый компилятор и встроенная поддержка асинхронных компонентов упрощают работу с серверными данными.
TypeScript 5.7 включён со строгим режимом проверки типов; типы доменной модели импортируются из общего workspace-пакета @dobrokot/shared, что гарантирует синхронизацию контрактов API между frontend и backend.
Tailwind CSS 4 с плагином @tailwindcss/typography используется как основной инструмент стилизации. Дизайн-токены (цвета, типографика, отступы) описаны через директиву @theme в globals.css, что позволяет переиспользовать переменные оформления в утилитарных классах и в произвольных CSS-блоках.
TanStack Query (React Query) 5 — клиент-кэш и менеджер серверных мутаций. Используется для всех POST-запросов с пользовательским вводом (проверка наличия крови, поиск клиник, активация подписки, отправка заявки на партнёрство).
Axios 1.x — HTTP-клиент с настраиваемыми interceptors, через которые реализованы добавление Bearer-токена и автоматическое обновление access-токена при ответе 401.
Radix UI (@radix-ui/react-checkbox, @radix-ui/react-tabs, @radix-ui/react-scroll-area, @radix-ui/react-slot) — набор unstyled-примитивов, на их основе построены доступные интерактивные элементы интерфейса с поддержкой клавиатуры и screen reader.
class-variance-authority и tailwind-merge — управление вариантами компонентов (например, размер и тема кнопки) и безопасное объединение Tailwind-классов с переопределением.
Leaflet 1.9 + react-leaflet 5 — карта доноров на странице /map, рендерит маркеры клиник поверх OpenStreetMap-тайлов.
lucide-react — пакет SVG-иконок, используемых в кнопках, бейджах и навигации.
zod — runtime-валидация на формах и тех же схемах, которые использует backend, переиспользована из @dobrokot/shared.

7.2 Архитектура файловой структуры
Исходный код frontend организован по слоистой структуре, близкой к Feature-Sliced Design, и расположен в каталоге apps/web/src. Структура состоит из следующих верхних каталогов.
•	app/ — маршруты Next.js App Router. Каждый подкаталог — это сегмент URL; файл page.tsx определяет страницу, layout.tsx — обёртку, providers.tsx — клиентскую часть провайдеров, globals.css — глобальные стили и дизайн-токены.
•	views/ — представления страниц, каждое в своём подкаталоге (views/main, views/blood-search, views/partnership, views/profile, views/clinic, views/plans, views/results, views/map, views/blog, views/signup, views/signin). Файл уровня каталога (например, views/profile/ProfilePage.tsx) описывает страницу в целом, подкаталог ui/ — её внутренние секции и карточки.
•	features/ — функциональные модули, не привязанные к конкретной странице, но реализующие отдельный пользовательский сценарий. На текущий момент это features/auth (хук useAuth и инфраструктура хранения токенов) и features/blood-search (универсальная форма BloodCheckForm и модалка BloodResultModal, переиспользуемые на главной и на странице поиска).
•	shared/ — переиспользуемые ресурсы. shared/ui содержит дизайн-системные примитивы (Badge, Button, Checkbox, Input, Modal, Select, ScrollArea, Tabs); shared/layout — компоненты глобальной разметки (Header, MobileMenu, Footer, QuestionsForm, BodyBackgroundController).
•	api/ — клиентские модули, инкапсулирующие работу с backend (auth.ts, blood.ts, subscription.ts, donors.ts, partnership.ts, geocoding.ts) поверх общего http.ts.
•	types/ — глобальные TypeScript-декларации (расширения для Leaflet, типы JSX-окружения).
Такая структура разделяет ответственности по слоям: app/ занимается только маршрутизацией, views/ собирают страницы из секций, features/ инкапсулируют сценарии, а shared/ предоставляет инструменты, на которые они опираются. Зависимости направлены строго сверху вниз — views/ могут импортировать из features/ и shared/, features/ — только из shared/ и api/; shared/ не зависит ни от чего, кроме внешних библиотек и @dobrokot/shared.

7.3 Маршрутизация и реализованные страницы
Маршруты конфигурируются файловой структурой каталога app/ согласно конвенции Next.js App Router. На текущий момент в платформе реализованы следующие маршруты.
•	/ — главная страница (views/main/MainPage). Включает два варианта герой-секции (мобильная/десктоп), блок «О нас», блок «Как это работает», карусель партнёров и секцию проверки наличия крови с динамической формой и модальным окном результата.
•	/signup — регистрация (views/signup/SignupPage + ui/SignupForm). Форма имени/фамилии/телефона/email/пароля с подтверждением, согласие на обработку данных, валидация на стороне клиента, отправка на POST /api/auth/register с автоматической авторизацией после успешного ответа.
•	/signin — вход (views/signin/SigninPage + ui/SigninForm). Email + пароль с переключаемой видимостью, отправка на POST /api/auth/login.
•	/profile — личный кабинет владельца животного (views/profile/ProfilePage). Две карточки: профиль владельца (PATCH /api/users/me на сохранение) и профиль питомца (хранится в localStorage, без бизнес-логики на backend — таково задание).
•	/clinic — личный кабинет клиники (views/clinic/ClinicPage). Доступен пользователям с ролью CLINIC, содержит редактирование контактов клиники и заглушки для прикрепления логотипа и CSV-файла доноров.
•	/plans — страница тарифов (views/plans/PlansPage). Два тарифа (Базовый — неделя, Профессиональный — месяц); по нажатию кнопки оплаты вызывается POST /api/subscription/activate (фейковая оплата согласно ТЗ — реальный платёжный шлюз не подключается), после успеха происходит редирект на /plans/success.
•	/plans/success — страница успешной оплаты тарифа (views/plans/PaymentSuccessPage).
•	/blood-search — поиск клиник с подходящей кровью (views/blood-search/BloodSearchPage). Доступен только авторизованным пользователям с активной подпиской; форма параметров + сетка карточек найденных клиник с дистанцией и кнопкой «Обратиться в клинику».
•	/partnership — страница партнёрства (views/partnership/PartnershipPage). Сетка из трёх интерактивных карточек и одной декоративной (3 + 1 в 4-колоночной сетке), форма заявки PartnershipForm с отправкой на POST /api/partnership/requests, секция партнёров.
•	/results — публичная витрина «Наши результаты» (views/results/ResultsPage) с карточками доноров и бейджем ранга (LEGENDARY_DONOR — оранжевый, RELIABLE_ASSISTANT — голубой).
•	/map — карта доноров на Leaflet (views/map/MapPage + ui/MapSection, DonorCard, SelectedDonorMap).
•	/blog — список публикаций (views/blog/BlogPage + ui/ArticleCard).
•	/blog/[id] — страница отдельной статьи (динамический сегмент).
•	/api/auth/[...nextauth] — единственный серверный route handler в app/api/, оставлен от next-auth, в текущей реализации не используется в качестве основного механизма (см. 7.7).

7.4 Корневой layout, провайдеры и глобальные стили
Файл app/layout.tsx определяет корневую разметку. Он подключает шрифт Manrope с поддержкой кириллицы через next/font/google (фоновое самостоятельное хостинг-проксирование Google Fonts), задаёт теги <html lang="ru"> и <body> с tailwind-классами фонового цвета и минимальной высоты, оборачивает контент в провайдеры и фиксированный контейнер шириной не более 1440 пикселей (max-w-8xl), а также подключает четыре глобальных компонента:
•	BodyBackgroundController — клиентский компонент, который в зависимости от текущего маршрута устанавливает CSS-переменную body bg (оранжевый на не-главной странице, серый на главной). Реализован через usePathname() и манипуляции с CSS на корневом элементе.
•	Header — фиксированная навигация, меняющая внешний вид в зависимости от состояния авторизации и наличия активной подписки (см. 7.7 и 7.9).
•	QuestionsForm — глобальный блок «Остались вопросы?» с формой обратной связи, видимый на всех страницах.
•	Footer — подвал с навигацией, ссылками на блог, иконками социальных сетей и кнопкой «Найти донора».
Файл app/providers.tsx — отдельный Client Component с директивой "use client" — инициализирует QueryClient (TanStack Query) с настройками retry: 1 и refetchOnWindowFocus: false и оборачивает дерево в QueryClientProvider. Подобное разделение нужно, потому что сам RootLayout остаётся Server Component, а провайдеры по своей природе требуют клиентского окружения.
Файл app/globals.css содержит:
•	импорты Tailwind v4 и плагина typography;
•	блок @theme с дизайн-токенами (цвета background, foreground, accent-orange, accent-sky, secondary, muted, typography, stroke, typography-inactive; кастомный размер --spacing-8xl: 1440px для контейнера максимальной ширины);
•	@layer base с базовыми правилами (применение фона и шрифта к body, cursor: pointer на кнопках);
•	keyframes-анимации (бесконечная карусель партнёров и появление карточек статей с поддержкой prefers-reduced-motion);
•	утилитарные правила для встроенной карты Leaflet (стили попапов и маркеров доноров).

7.5 Дизайн-система и UI-примитивы
В каталоге shared/ui собраны базовые компоненты, на которых строится визуальный язык платформы. Все они написаны с использованием class-variance-authority для управления вариантами и tailwind-merge для безопасного объединения классов.
Button (shared/ui/button.tsx) — кнопка с четырьмя вариантами (default — тёмный фон, white — белый, outline — обводка тёмного цвета, outlineWhite — белая обводка) и тремя размерами (default, icon, iconLg). Поддерживает паттерн asChild через @radix-ui/react-slot, что позволяет применять стили кнопки к произвольному элементу (например, ссылке) без вложенности.
Badge (shared/ui/badge.tsx) — пилюля-метка с четырьмя вариантами (default — оранжевая, sky — голубая, transparent — стеклянная с блюром, biege — бежевая) и двумя размерами. Используется для статусов и подсказок.
Input (shared/ui/input.tsx) — текстовое поле с двумя визуальными вариантами (default — карточная рамка, form — подчёркивание для inline-форм) и двумя размерами. Поддерживает все стандартные input-атрибуты, плюс отдельный htmlSize для нативного size, чтобы избежать конфликта с CVA-размером.
Checkbox (shared/ui/checkbox.tsx) — обёртка над @radix-ui/react-checkbox с иконкой Check из lucide-react.
Modal (shared/ui/modal.tsx) — модальное окно с затемнением фона, закрытием по Escape, блокировкой скролла body и адаптивной шириной. Используется в результатах проверки крови и при подтверждении действий.
Select (shared/ui/select.tsx) — стилизованный нативный <select> со встроенной иконкой ChevronDown, выбор вида животного, группы крови, пола питомца.
ScrollArea и Tabs — обёртки над соответствующими Radix-примитивами.
В каталоге shared/layout расположены крупные структурные компоненты:
•	Header — фиксированный хедер. Содержит логотип (меняется на белый вариант на не-главной), горизонтальную навигацию для десктопа, кнопку «Найти донора» (поведение зависит от состояния авторизации и подписки), иконку профиля (только для авторизованных) и переключатель мобильного меню. Цвет фона меняется на оранжевый, если пользователь авторизован или открыта не главная.
•	MobileMenu — выпадающее мобильное меню с теми же ссылками плюс «Личный кабинет» и «Поиск крови» (если есть подписка).
•	Footer — двух- и одноколоночная (на мобильной и десктопе) разметка с тёмным фоном, повторяющая структуру навигации.
•	QuestionsForm — блок обратной связи с формой имени, email, телефона и сообщения.
•	BodyBackgroundController — побочный эффект для управления глобальным фоном.

7.6 HTTP-клиент и интеграция с API
Все обращения к backend проходят через единственный экземпляр axios, описанный в api/http.ts. Базовый URL берётся из переменной окружения NEXT_PUBLIC_API_URL (по умолчанию http://localhost:4000/api), заголовок Content-Type выставляется в application/json.
К клиенту подключены два interceptor.
Request interceptor — на каждый исходящий запрос читает access-токен из localStorage (ключ dobrokot:accessToken) и при наличии добавляет заголовок Authorization: Bearer <token>.
Response interceptor — реализует автоматический refresh при истечении access-токена. При ответе 401 на любом эндпоинте, кроме /auth/*, перехватчик помечает оригинальный запрос флагом _retry, читает refresh-токен из localStorage, отправляет POST /auth/refresh, и при успешном ответе записывает новую пару токенов в localStorage и повторяет исходный запрос с обновлённым заголовком. Запросы внутри одного «окна обновления» используют общий промис refreshPromise, чтобы избежать гонок при параллельных запросах. Если refresh не удался, токены очищаются — пользователь автоматически считается анонимным.
Поверх axios построены тонкие сервисные модули.
•	api/auth.ts — registerUser, loginUser, logoutUser. Каждая успешная функция, кроме logout, сохраняет полученную пару токенов через tokenStorage.set.
•	api/blood.ts — checkBlood (POST /blood/check, публичный), searchBlood (POST /blood/search, требует подписки) и константа bloodTypeOptionsByAnimal, отображающая enum-значения групп крови в человекочитаемые подписи.
•	api/subscription.ts — fetchSubscription (GET /subscription/me) и activateSubscription (POST /subscription/activate).
•	api/donors.ts — getDonors с парой query-параметров и одноимённый хук useDonorsQuery, обёрнутый в useQuery.
•	api/partnership.ts — submitPartnershipRequest.
•	api/geocoding.ts — вспомогательные обращения к геокодеру для страницы карты.
Утилита unwrapApiResponse разбирает стандартный конверт { success, data, error?, message? }, который возвращает backend: при success=false бросает ошибку с текстом из поля error/message, иначе возвращает поле data в типобезопасной форме.

7.7 Аутентификация и состояние пользователя
Центральным элементом клиентской авторизации является хук useAuth (features/auth/useAuke.ts). Он реализует один глобальный store без сторонних библиотек state-management.
Хранилище реализовано через React API useSyncExternalStore: модуль владеет приватной переменной currentState и набором подписчиков listeners; функция setState мутирует состояние и оповещает всех подписчиков. Такой подход сохраняет совместимость с конкурентным рендером React 19 и не требует отдельного провайдера.
Состояние включает: isAuthenticated, user (id/email/name/role), subscription (SubscriptionState с полями plan, expiresAt, isActive), pet (профиль питомца, хранится в localStorage), hydrated (флаг завершённой инициализации).
Гидратация происходит однократно при первом монтировании любого потребителя: если в localStorage есть access-токен, выполняются параллельно GET /users/me и GET /subscription/me; результаты записываются в состояние. Если запросы провалились — токены очищаются, пользователь становится анонимным. Флаг hydrated предотвращает мерцание интерфейса (например, редиректы из защищённых страниц происходят только после hydrated=true).
Хук возвращает методы:
•	signUp(payload) — регистрация + автоматическая авторизация;
•	signIn(email, password) — вход;
•	signOut() — выход с очисткой токенов и состояния;
•	purchasePlan(plan: SubscriptionPlan) — активация тарифа через POST /subscription/activate;
•	refreshSubscription() / refreshUser() — повторное чтение профиля и подписки;
•	updatePet(pet) — мутация локального профиля питомца.
Файл api/http.ts экспортирует объект tokenStorage с методами getAccess/getRefresh/set/clear — это единая точка работы с localStorage, чтобы при изменении схемы хранения (например, переход на cookie) код useAuth остался прежним.
Next-auth изначально планировался как механизм авторизации (в каталоге app/api/auth остался route handler), но в итоговой реализации был заменён на собственный JWT-based flow с refresh-токенами — он точнее соответствует архитектуре backend, где /api/auth/* возвращает пару токенов с подписью и сроком жизни, а не сессионный cookie.

7.8 Работа с данными через TanStack Query
Все взаимодействия с API, кроме чисто навигационных, реализованы через React Query. На уровне shared-провайдеров создаётся QueryClient с настройками retry: 1 и refetchOnWindowFocus: false (то и другое снижает нагрузку на API в плохих сетевых условиях и избавляет от лишних запросов при возврате во вкладку).
Для запросов чтения используется useQuery: пример — useDonorsQuery в api/donors.ts, который описывает ключ ["donors", params], normalizes параметры и возвращает PaginatedResponse<Donor>. Этот хук применяется на странице карты.
Для мутаций используется useMutation. Каждая страница, отправляющая данные на backend, описывает свою мутацию и связывает с ней состояние интерфейса:
•	BloodCheckSection на главной — мутация checkBlood, по успеху открывает модалку BloodResultModal с вариантом found/not-found в зависимости от поля available в ответе;
•	BloodSearchPage — мутация searchBlood, по успеху сохраняет страницу результатов в локальное состояние; обрабатываются коды 401 (не авторизован) и 402 (нет подписки) с показом подсказок и ссылок на /signin и /plans соответственно;
•	PlansPage — мутация на активацию тарифа с редиректом на /plans/success;
•	PartnershipForm — мутация на отправку заявки;
•	SignupForm и SigninForm — мутации на регистрацию и вход.
Универсальная утилита extractApiError, повторяющаяся в каждом потребителе, извлекает текст ошибки из axios-ошибки (поля error/message в data) или из стандартного Error и показывает его пользователю.

7.9 Ключевые пользовательские сценарии
Сценарий «Проверить наличие крови». На главной странице пользователь заполняет форму BloodCheckForm (вид животного, порода, возраст, вес, группа крови, пол) и нажимает «Проверить наличие». На клиенте формируется payload вида { animalType, bloodType }, отправляется на публичный POST /api/blood/check (авторизация не требуется), backend возвращает { available, matchCount, clinicsCount }. По ответу открывается модальное окно: при available=true показывается вариант «Нашли для Вас N клиник с донорами» с кнопкой перехода на /blood-search, иначе — вариант «К сожалению, сейчас в клиниках нет крови нужной Вам группы» с кнопкой «Попробовать найти другую группу крови», закрывающей модалку.
Сценарий «Регистрация и оплата тарифа». Пользователь регистрируется (frontend получает пару токенов, сохраняет их в localStorage и обновляет состояние useAuth), переходит на /plans, выбирает тариф и нажимает «Оплатить». Frontend вызывает POST /api/subscription/activate с выбранным планом; на стороне backend подписка реально активируется (выставляется subscriptionExpiresAt на 7 или 30 дней вперёд), но платёжный шлюз не подключается — это сознательное упрощение согласно ТЗ. После успеха пользователь редиректится на /plans/success, состояние useAuth обновляется (поле subscription.isActive становится true), а Header автоматически перерисовывается: появляется пункт меню «Поиск крови», кнопка «Найти донора» начинает вести на /blood-search.
Сценарий «Поиск крови по подписке». На странице /blood-search пользователь заполняет ту же форму, что и на главной (компонент BloodCheckForm переиспользуется); по «Найти» отправляется POST /api/blood/search; в ответе приходит пагинированная коллекция клиник, отсортированная по расстоянию или количеству совпавших единиц крови. Для каждой клиники в выдаче показываются её название, адрес, рабочие часы, число подходящих единиц и расстояние; по кнопке «Обратиться в клинику» происходит переход на tel: с указанным телефоном.
Сценарий «Заявка на партнёрство». На странице /partnership любая из трёх интерактивных карточек открывает форму PartnershipForm, которая отправляет данные на публичный POST /api/partnership/requests без авторизации.
Сценарий «Профиль владельца и питомца». На /profile владелец редактирует свои контактные данные (имя/фамилия/телефон) — изменения уходят на PATCH /api/users/me, после чего useAuth.refreshUser() читает обновлённого пользователя. Профиль питомца (кличка, вид, порода, возраст, вес, группа крови, пол) сохраняется в localStorage без обращения к backend — таково задание.

7.10 Карта доноров
Страница /map отображает карту OpenStreetMap с маркерами клиник-доноров, используется библиотека Leaflet и обёртка react-leaflet. Сами компоненты разнесены: views/map/MapPage — корневая страница; views/map/ui/MapSection — карта с центром, зумом и маркерами; views/map/ui/DonorCard — карточка выбранного донора, появляющаяся над картой; views/map/ui/SelectedDonorMap — мини-вариант карты с подсветкой одного маркера.
Поскольку Leaflet требует доступа к window, компонент карты помечен директивой "use client" и при необходимости подгружается через dynamic-импорт без SSR. Типы leaflet и react-leaflet расширены в types/leaflet.d.ts и types/react-leaflet.d.ts.
Стили Leaflet импортируются в globals.css (@import "leaflet/dist/leaflet.css"), а кастомизация маркеров (donor-marker-icon) и попапов вынесена в утилитарные CSS-правила того же файла.

7.11 Адаптивность и производительность
Дизайн-система выстроена под mobile-first: базовые классы рассчитаны на мобильный экран, варианты для desktop включаются через брейкпойнт sm: (640px и выше). Корневой контейнер ограничен 1440 пикселями (max-w-8xl) — на широких мониторах контент выровнен по центру, фоновый цвет body заполняет область за его пределами. Это позволяет смотреть лендинг и кабинет одинаково комфортно как на смартфоне, так и на FullHD-мониторе.
Изображения подаются через next/image: автоматическая оптимизация (AVIF/WebP), ленивая загрузка, корректные размеры и srcset под разные плотности экрана. Для критичных изображений (например, большая декоративная карточка на странице партнёрства) включён приоритет загрузки через priority.
Шрифт Manrope подключается через next/font/google, что устраняет flash of unstyled text: подмножество glyphs запекается в статический файл на этапе сборки.
Все интерактивные компоненты (формы, модалки, кабинеты) помечены директивой "use client" и попадают в client-bundle, тогда как статические секции лендинга остаются Server Components и не отгружают лишнего JS на клиент.
В TanStack Query настроена дедупликация одинаковых запросов и кэш с staleTime 30 секунд для списка доноров — повторное открытие страницы /map не приводит к лишнему сетевому запросу.

7.12 Значение клиентской части в общей системе
Frontend играет роль входной точки и слоя представления для всех бизнес-сценариев платформы: проверка наличия крови, регистрация владельцев и клиник, оформление и оплата тарифа, поиск контактов клиник по подписке, заполнение профиля питомца, отправка заявок на партнёрство, отображение публичной витрины результатов и карты доноров. Все эти функции были реализованы в рамках работы и интегрированы с backend через единый типобезопасный контракт, описанный в пакете @dobrokot/shared. Таким образом, клиентская часть не сводится к набору демонстрационных страниц — она представляет собой полнофункциональную пользовательскую часть продукта, готовую к промышленной эксплуатации в связке с backend.
